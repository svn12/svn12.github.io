為什麼過去大家用 Markdown？

早期 LLM 的 context window 只有幾 K token，開發者必須節省每一個字元。 Markdown 語法簡潔，HTML 標籤比 Markdown 多消耗大量 token，在小 context 時代不划算。

HTML 派的四大主張：解決「認知瓶頸」

破除認知瓶頸
當 AI 生成數千行 Markdown 規格書或計畫時，人類容易流於橡皮圖章，盲目點擊同意。HTML 則能強迫專注
高視覺資訊密度
透過 CSS 雙欄排版、卡片、SVG 流程圖等，將線性文字轉為空間佈局，讓人能「一眼看懂」複雜結構
丟棄式微 UI (Throwaway UI)
內嵌 JS。AI 不只是寫文章，更是 App 開發者。可生成含按鈕/表單的微型 UI，修改後直接匯出
分享便利性
單一包含樣式與邏輯的 .html 檔案可在任何瀏覽器直接開啟，極度利於團隊與跨部門分享與決策

核心主張

Markdown 正在成為「沒有人讀 AI 產出內容」的原因之一——
它讓 AI 的輸出停留在線性文字的層次，
而 HTML 能讓 AI 輸出成為真正的多維度互動文件。
— Thariq Shihipar，Anthropic Claude Code 工程師
HTML 可在單一自我包含的文件中同時呈現：
複雜表格、SVG 向量插圖、可執行程式碼、
JavaScript 互動元件、空間佈局，以及文件範圍樣式表。
— AI 工程師社群共識

Markdown 守護派的反駁 ⚠️
  • ✦ Token 成本暴增：HTML 標籤消耗的 Token 是 Markdown 的 2 到 4 倍
  • ✦ Git Diff 災難（雜訊過多）：代碼變更夾雜大量 CSS/標籤，導致 code review 極難閱讀
  • ✦ 機器與 RAG 相容性：AI 在檢索與理解 Markdown（特別是表格）的精準度仍高於 HTML

RAG 場景的例外

2026 年 benchmark 研究顯示：在 RAG（檢索增強生成）的資料處理管線中，
Markdown 仍有優勢；但在最終展示端輸出，HTML 明顯勝出。

結論：HTML 與 Markdown 並非零和競爭，適用場景不同

結論與建議

選 HTML 的時機
  • AI Agent 最終輸出報告
  • 需要視覺化或圖表
  • 分享給非技術人員
  • 互動式儀表板
  • 長篇多章節文件
選 Markdown 的時機
  • 需要 Git 版本控制
  • RAG 資料處理管線
  • 開發者協作文件
  • 需要多格式轉換
  • 純文字編輯場景

🎯 工程共識：機器/代碼庫維持使用 Markdown (Git 友善、Token 效率高)
人類/交付階段使用 HTML (作為人機互動、雙向協作與打破認知瓶頸的橋樑)
